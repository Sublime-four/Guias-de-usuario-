Guía WMS — Estados de Albaranes (stock.picking) vía REST API

Esta guía documenta cómo el WMS puede consultar y devolver los estados de los albaranes (transferencias / stock.picking) de Odoo 17/18 mediante la REST API expuesta por el módulo restful.

Alcance: esta documentación cubre exclusivamente la gestión del campo state de stock.picking: cómo consultarlo, cómo cambia automáticamente según las acciones invocadas desde WMS, y cómo reflejar ese estado de vuelta hacia el WMS.

1. Ciclo de vida del estado (state)

El campo state de stock.picking es una selección controlada por Odoo. No debe escribirse directamente vía PUT; su valor cambia como consecuencia de las acciones invocadas (por ejemplo action_confirm, action_assign, button_validate) o por reglas internas del módulo stock.

Valor (state)EtiquetaSignificado
draftBorradorPicking recién creado, sin confirmar. Editable.
waitingEn espera de otra operaciónDepende de otro movimiento que aún no ha terminado (flujos encadenados).
confirmedEsperando disponibilidadConfirmado, pero sin stock reservado todavía.
assignedListo (reservado)Stock reservado, listo para alistar/despachar/recibir.
doneHechoPicking validado; inventario actualizado.
cancelCanceladoPicking cancelado; no afecta inventario.

2. Autenticación

GET /api/auth/token

GET https://<odoo_host>/api/auth/token?login=<user>&password=<api_key>&db=<db_name>

Incluir en todas las llamadas siguientes el header access_token: <token> (guion bajo).

3. Consultar el estado de un albarán

3.1 Por ID

GET /api/stock.picking/<id>

curl -X GET "https://<odoo_host>/api/stock.picking/123" \
  -H "access_token: <token>"

Respuesta (referencial, campos relevantes):

{
    "count": 1,
    "data": [
        {
            "id": 123,
            "name": "WH/OUT/00001",
            "state": "assigned",
            "picking_type_code": "outgoing",
            "origin": "S00001",
            "scheduled_date": "2026-04-24 10:00:00",
            "date_done": false
        }
    ]
}

3.2 Por referencia externa (origin)

curl -X GET "https://<odoo_host>/api/stock.picking/" \
  -H "access_token: <token>" \
  -d "limit=1" \
  -d "domain=[('origin','=','WMS-OUT-000123')]"

3.3 Listar albaranes filtrando por estado

curl -X GET "https://<odoo_host>/api/stock.picking/" \
  -H "access_token: <token>" \
  -d "limit=50" -d "offset=0" \
  -d "order=create_date desc" \
  -d "domain=[('state','in',['assigned','confirmed']),('picking_type_code','=','outgoing')]"

4. Acciones que modifican el estado

Las siguientes acciones cambian el state del picking como efecto colateral. El WMS debe invocarlas en el momento oportuno y consultar nuevamente el picking para obtener el estado resultante.

AcciónEndpointTransición de estado
action_confirm POST /api/stock.picking/<id>/action_confirm draft → confirmed (o waiting)
action_assign POST /api/stock.picking/<id>/action_assign confirmed → assigned (si hay stock)
button_validate POST /api/stock.picking/<id>/button_validate assigned → done
action_cancel POST /api/stock.picking/<id>/action_cancel cualquiera → cancel

Ejemplo:

curl -X POST "https://<odoo_host>/api/stock.picking/123/button_validate" \
  -H "access_token: <token>"
Importante: el campo state es read-only desde la API. No se puede escribir con PUT /api/stock.picking/<id>. Si WMS envía {"state": "done"} el cambio será ignorado o rechazado por Odoo.

5. Patrón recomendado para WMS: “devolver el estado”

Tras cada interacción con Odoo, WMS debe recuperar y registrar el estado resultante:

  1. Ejecutar la acción (button_validate, action_assign, etc.).
  2. Hacer GET /api/stock.picking/<id> inmediatamente después.
  3. Leer los campos state, date_done, name y backorder_id.
  4. Reportar ese estado hacia los sistemas internos del WMS.
# 1. Validar
curl -X POST "https://<odoo_host>/api/stock.picking/123/button_validate" \
  -H "access_token: <token>"

# 2. Consultar estado resultante
curl -X GET "https://<odoo_host>/api/stock.picking/123" \
  -H "access_token: <token>"

6. Campos útiles asociados al estado

CampoTipoDescripción
stateSelectionEstado actual del albarán (ver sección 1).
date_doneDatetimeFecha de validación. Sólo se informa cuando state = done.
scheduled_dateDatetimeFecha planificada.
backorder_idMany2oneSi al validar se generó un backorder (faltantes), referencia al picking hijo.
picking_type_codeSelectionTipo del albarán (incoming, outgoing, internal).
originCharReferencia externa para correlacionar con el WMS.
move_ids / move_line_idsOne2manyLíneas y movimientos; también poseen su propio state.

7. Manejo de errores

HTTPCausa comúnAcción recomendada (WMS)
401Token inválido / expirado / header mal escritoRegenerar token. Verificar header access_token (guion bajo).
403Usuario sin permisos sobre stock.pickingAjustar grupos/permisos del usuario API.
404Picking no existe (ID inválido o eliminado)Re-sincronizar IDs; buscar por origin.
400 / 422Estado incompatible con la acción (ej. validar un draft)Verificar precondiciones; invocar action_confirm y action_assign antes de button_validate.
500Error interno de OdooLog + reintento con backoff; escalar si persiste.

8. Checklist mínimo de integración de estados

  • Consumir GET /api/stock.picking/<id> después de cada acción para leer state.
  • Persistir en el WMS la última fecha/estado observado y su origen (origin).
  • Nunca asumir transición; siempre re-consultar Odoo.
  • No intentar escribir state directamente con PUT.
  • Correlacionar siempre por origin cuando haya reintentos.
